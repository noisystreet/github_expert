团队管理
========

上一章我们学习了工作流——Pull Request、代码审查、CI/CD。这些工具让协作更顺畅，但协作的前提是有效的团队管理：谁可以修改代码、谁可以合并 PR、如何分配任务。

GitHub 提供了两种协作模式：个人仓库的协作者模式，和组织仓库的团队模式。理解这两种模式的差异，能帮助你选择合适的协作方式。

两种协作模式
------------

**个人仓库的协作者模式**

如果你创建了一个个人仓库，可以添加协作者。打开仓库的 Settings → Collaborators，输入对方的 GitHub 用户名，发送邀请。对方接受后，就成为仓库的协作者。

协作者可以推送代码、创建 Issue、评论 PR。但你（仓库所有者）可以控制协作者的权限级别：

.. list-table:: 协作者权限级别
   :widths: 20 80
   :header-rows: 1

   * - 权限
     - 能做什么
   * - Read
     - 查看代码、克隆仓库、创建 Issue
   * - Triage
     - 管理 Issue（添加标签、关闭），但不能推送代码
   * - Write
     - 推送代码、合并 PR、管理 Release
   * - Maintain
     - 管理仓库设置、创建分支保护规则
   * - Admin
     - 完全控制，包括删除仓库、转移所有权

对于小团队，协作者模式足够。但如果团队有几十人，管理协作者变得繁琐。这时应该使用组织模式。

**组织仓库的团队模式**

组织（Organization）是 GitHub 提供的团队管理单元。创建组织后，可以创建多个团队，每个团队有自己的权限。

组织的优势：

- **团队分组**：按部门或项目分组，如 "frontend-team"、"backend-team"
- **统一权限**：团队内所有成员权限一致，无需逐个设置
- **仓库共享**：组织创建的仓库所有成员可见，私有仓库也可共享
- **成本优化**：组织可以付费购买高级功能，按团队计费

创建组织：GitHub 主页右上角 "+" → "New organization"。填写组织名称，邀请成员。

组织的管理层级：

- **Owners**：组织所有者，可以管理成员、创建仓库、设置权限
- **Members**：组织成员，可以访问组织的仓库（权限由团队设置决定）
- **Outside collaborators**：外部协作者，不属于组织，但对特定仓库有权限

团队的权限设置
--------------

在组织中，权限通过团队管理。打开组织页面 → Teams，创建团队。团队可以分配仓库的权限级别：

.. list-table:: 团队权限级别
   :widths: 20 80
   :header-rows: 1

   * - 权限
     - 团队能做什么
   * - Read
     - 只读，适合 QA、设计师等非开发角色
   * - Write
     - 开发权限，适合开发团队
   * - Maintain
     - 维护权限，适合项目负责人
   * - Admin
     - 管理权限，适合团队管理员

**权限继承**

团队可以嵌套。比如 "engineering" 团队包含 "frontend" 和 "backend" 子团队。父团队的权限会被子团队继承。

这种结构适合大型组织。顶层 "engineering" 团队有 Read 权限，"frontend" 子团队有 Write 权限。这样 frontend 成员既能访问所有 engineering 仓库，又能推送 frontend 相关仓库。

**案例：一个团队的权限配置**

假设一个项目有三个团队：

- Core team（核心开发）：Admin 权限，可以合并 PR、管理 Release
- Dev team（普通开发）：Write 权限，可以推送代码、创建 PR
- QA team（测试团队）：Read 权限，可以查看代码、创建 Issue

这种配置确保核心开发者有管理权限，普通开发者有开发权限，测试团队只有查看权限。职责清晰，权限分明。

项目管理工具
------------

GitHub 提供了多个项目管理工具，帮助团队协作。

**Projects（项目看板）**

Projects 是任务看板，类似 Trello。可以创建列（如 "To do"、"In progress"、"Done"），把 Issue 和 PR 添加到列中。

创建项目：仓库页面的 "Projects" 标签 → "New project"。选择模板（Board、Roadmap 等）。

项目看板的用途：

- 规划任务：把待办事项添加到 "To do" 列
- 追踪进度：开发中的任务移到 "In progress" 列
- 标记完成：完成的任务移到 "Done" 列

项目可以关联多个仓库，适合跨仓库的项目管理。

**Milestones（里程碑）**

里程碑是时间节点，标记项目的阶段性目标。比如 "v1.0 Release"、"Beta Testing"。

创建里程碑：Issue 页面 → "Milestones" → "New milestone"。填写标题、描述、截止日期。

里程碑的用途：

- 规划发布：把 v1.0 要完成的 Issue 添加到里程碑
- 追踪进度：里程碑显示完成百分比
- 激励团队：明确的截止日期让团队有紧迫感

**Labels（标签）**

标签用于分类 Issue 和 PR。GitHub 默认提供一些标签：bug、documentation、enhancement 等。

自定义标签：Issue 页面 → "Labels" → "New label"。填写名称、颜色、描述。

常用的标签分类：

- 类型：bug（问题）、feature（功能）、documentation（文档）
- 优先级：priority-high、priority-low
- 状态：in-progress、blocked、ready-for-review
- 范围：frontend、backend、testing

标签帮助快速筛选 Issue。比如查看所有 "bug" 标签的 Issue，了解待修复的问题。

团队协作的最佳实践
------------------

**权限分配原则**

权限分配遵循最小权限原则：给成员完成任务所需的最小权限，不多给。

为什么？权限过大会带来风险：

- 误删分支或仓库
- 错误合并 PR
- 修改仓库设置影响他人

普通开发者只需要 Write 权限。维护者需要 Maintain 权限。管理员需要 Admin 权限。清晰分工，降低风险。

**责任分工**

团队应该明确责任分工。比如：

- 负责人：合并 PR、发布 Release、管理分支保护规则
- 开发者：提交代码、审查 PR、创建 Issue
- QA：测试代码、报告 Bug、验证修复

分工避免职责重叠。比如 PR 合并，如果所有人都能合并，可能出现随意合并的情况。规定只有负责人可以合并，确保合并前充分审查。

**沟通机制**

团队协作需要沟通。GitHub 提供了多种沟通渠道：

- Issue 讨论：在 Issue 中讨论问题和方案
- PR 评论：在 PR 中讨论代码改动
- Discussions：社区讨论，适合开放式讨论
- 团队页面：团队成员列表，可以发起团队讨论

建议重要决策在 Issue 或 Discussions 中讨论，记录讨论过程，方便回顾。

**新人引导**

新成员加入团队时，需要引导：

1. 阅读 README 和 CONTRIBUTING，了解项目
2. 配置开发环境，克隆仓库
3. 从小任务开始，如修复文档、添加测试
4. 参与代码审查，学习团队代码风格
5. 逐步承担更大的任务

建议为新人准备一份入门指南，包含项目介绍、环境配置、开发流程、常用命令。

组织的成本与选择
----------------

**组织的费用**

GitHub 组织有免费和付费版本：

- **Free**：免费，基本功能，适合小团队
- **Team**：每月 $4/用户，提供团队管理功能
- **Enterprise**：每月 $21/用户，提供高级安全和合规功能

免费版本对大多数团队足够。付费版本适合需要高级功能的企业。

**何时使用组织**

判断是否需要组织：

- 成员超过 5 人：组织管理更高效
- 多个仓库需要共享：组织仓库成员可见
- 需要团队分组：组织支持团队嵌套
- 需要高级权限控制：组织权限更灵活

如果只有 2-3 个协作者，个人仓库的协作者模式足够。如果团队超过 10 人，建议使用组织。

--------------------
