基本概念
========

上一节我们配置好了账户和 SSH 密钥，开发环境已准备就绪。现在，在动手操作之前，我们需要理解 GitHub 的三个核心概念：仓库、分支和提交。这些概念贯穿所有操作，理解它们能让你在遇到问题时快速定位原因。

仓库：代码的容器
----------------

仓库（Repository）是 GitHub 最基本的概念。它存储项目的所有文件，包括源代码、文档、配置文件，以及每个文件的完整变更历史。

为什么需要仓库？想象你正在开发一个应用，目录结构如下：

::

   my-app/
   ├── src/
   │   ├── main.py
   │   └── utils.py
   ├── tests/
   │   └ test_main.py
   ├── README.md
   └── requirements.txt

如果只用普通文件夹保存，你只能看到当前的代码状态。如果上周的 ``main.py`` 有一个 Bug 修复，你想看看当时的代码，无从查起。如果同事改了 ``utils.py``，你也无从得知改动内容。

仓库解决了这些问题。它不仅存储文件，还存储每次变更的快照（提交）。你可以查看任何一个历史版本的文件，对比不同版本之间的差异，甚至恢复到某个历史版本。

**仓库的结构**

一个仓库包含：

- **工作目录**：你编辑的文件
- **暂存区（Stage）**：准备提交的变更
- **仓库历史**：已保存的提交记录

这三者的关系就像拍照：工作目录是场景，暂存区是取景框，提交是按下快门保存的照片。

.. code-block:: bash

   # 查看仓库状态（三者关系）
   git status

   # 输出示例：
   # Changes not staged for commit:  （工作目录有改动，未进入暂存区）
   #   modified: main.py
   # Changes to be committed:        （暂存区有改动，等待提交）
   #   new file: test_main.py
   # nothing added to commit          （暂存区和工作目录一致）

理解这个状态输出很重要。Git 的三个区域状态不一致时，``git status`` 会告诉你哪些文件在哪个区域，帮助你决定下一步操作。

分支：并行开发的利器
--------------------

分支（Branch）让多人协作变得可行。它的核心思想是：不同的人可以在不同的分支上独立工作，最后合并到一起。

为什么需要分支？假设你和同事同时在 ``main.py`` 上开发新功能。你改了一半，同事也改了一半，两份改动冲突。你不知道同事改了什么，同事也不知道你的计划。合并时只能手动解决冲突，耗时耗力。

分支把这种串行开发变成了并行开发：

::

   main ────●────●────●────●
            │
   feature  ●────●────●

你创建一个 ``feature`` 分支，在这个分支上开发。同事继续在 ``main`` 分支工作。两个分支独立演进，互不干扰。等你完成开发后，把 ``feature`` 分支合并回 ``main``。

**分支的本质**

分支本质上是一个指向某个提交的可移动指针。创建分支时，Git 只是创建了一个新指针，指向当前的提交。文件并没有复制，只是记录了一个新的工作起点。

.. code-block:: bash

   # 创建分支（创建指针）
   git branch feature/login

   # 切换分支（移动 HEAD 指针）
   git checkout feature/login

   # 创建并切换（常用组合）
   git checkout -b feature/login

   # 查看所有分支
   git branch -a

HEAD 是一个特殊指针，指向你当前所在的分支。切换分支就是移动 HEAD，让它指向不同的分支指针。文件内容随之切换到对应分支的状态。

**分支命名约定**

团队通常会约定分支命名规则：

- ``main`` 或 ``master``：主分支，生产环境代码
- ``develop``：开发分支，集成各功能
- ``feature/*``：功能分支，如 ``feature/login``
- ``bugfix/*``：修复分支，如 ``bugfix/login-error``
- ``release/*``：发布分支，如 ``release/v1.0``

命名规范让分支用途一目了然，协作时减少沟通成本。

提交：变更的原子单位
--------------------

提交（Commit）是 Git 记录变更的基本单位。每次提交保存一个快照，记录谁在什么时候改了什么内容。

为什么提交是原子单位？假设你修改了十个文件，如果一次性提交，出问题时很难定位。如果分成十个提交，每个提交只改一个文件，问题就容易定位：只需查看对应提交的改动。

提交的最佳实践：

**一个提交只做一件事**

- 修复 Bug：只包含 Bug 修复的改动
- 添加功能：只包含新功能相关的改动
- 重构代码：只包含重构相关的改动

不要把多种改动混在一个提交里。重构时顺便修复 Bug 看起来高效，但出问题时难以追溯。

**提交信息要清晰**

.. code-block:: bash

   # 好的提交信息
   git commit -m "Fix login validation error when username is empty"

   # 差的提交信息
   git commit -m "fix bug"
   git commit -m "update"
   git commit -m "changes"

好的提交信息回答三个问题：做了什么（动词），影响什么（对象），为什么（背景）。阅读提交历史时，清晰的提交信息能快速了解变更内容。

**提交的工作流程**

.. code-block:: bash

   # 1. 编辑文件
   # （在工作目录）

   # 2. 添加到暂存区
   git add main.py              # 单个文件
   git add .                    # 所有改动

   # 3. 查看暂存区状态
   git status

   # 4. 提交
   git commit -m "Add user authentication"

   # 5. 推送到远程
   git push origin main

这个流程是 Git 操作的核心。很多初学者困惑于 ``git add`` 的必要性：为什么不能直接 ``git commit``？

答案在于控制粒度。``git add`` 让你选择哪些改动进入本次提交。你可能改了十个文件，但只想提交其中五个。暂存区给你选择的机会。

概念之间的关系
--------------

仓库、分支、提交三者构成 Git 的核心模型：

- 仓库：包含所有分支和提交历史
- 分支：指向某个提交，是开发的工作线
- 提交：是变更的快照，是历史的节点

理解这个模型，后续的操作就有了意义。克隆仓库是获取完整的模型副本，切换分支是移动到不同的工作线，推送代码是同步本地模型到远程。
