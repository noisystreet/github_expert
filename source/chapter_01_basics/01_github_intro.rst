GitHub 简介
===========

你刚写完一段代码，想保存起来。最简单的方式是复制一份，改名为 ``code_v2.py``。下次修改再复制成 ``code_v3.py``。一个月后，文件夹里堆满了 ``code_final.py``、``code_final_final.py``、``code_真的最终版.py``。

这不是笑话，这是很多开发者真实的经历。版本管理的需求由此而生——我们需要一种方式，既能保存代码的变更历史，又能随时回溯到任意版本。

Git 解决了这个问题。它是 Linus Torvalds 在 2005 年为管理 Linux 内核开发而创建的分布式版本控制系统。

.. note::
   **Git 的诞生故事**
   
   2005 年，Linux 内核社区使用的版本控制工具 BitKeeper 收回了免费授权。Linus Torvalds 只有两周时间寻找替代方案。他试用了几个现有工具（CVS、SVN），都不满意——要么太慢，要么不支持分布式协作。
   
   于是 Linus 决定自己写一个。他设计了 Git 的核心特性：快、简单、支持非线性开发（成千上万的并行分支）、完全分布式。Git 的第一版代码在 2005 年 4 月 7 日发布，只用了两周开发，C 代码量不到 1000 行。
   
   今天，Git 是全球最流行的版本控制系统，超过 90% 的开发者使用 Git。Linus 后来调侃："我是个懒惰的开发者，所以我写了 Git。"

但 Git 是一个本地工具，代码只存在你的电脑上。如果硬盘坏了，代码就丢了。如果要和别人协作，只能互相发送文件，手动合并修改。

这就是 GitHub 出现的原因。GitHub 在 2008 年上线，核心想法很简单：把 Git 仓库放到云端，提供一个统一的托管平台。

.. tip::
   **GitHub 的诞生**
   
   GitHub 由三个开发者创立：Tom Preston-Werner、Chris Wanstrath、PJ Hyett。2008 年 4 月，他们在旧金山的一家咖啡馆讨论想法：为什么没有一个好的 Git 仓库托管网站？
   
   当时已有类似服务（如 SourceForge、Google Code），但它们基于 SVN 或 CVS，不支持 Git。三人决定做一个"周末项目"，一周内上线了 GitHub 的原型。
   
   GitHub 第一个仓库是 Tom 的个人项目 "Grit"，一个 Ruby 的 Git 库实现。第一个外部用户是 Ruby on Rails 框架，他们在 2008 年 4 月迁移到 GitHub，带来大量 Ruby 开发者。
   
   2008 年，GitHub 只有几千用户。今天，GitHub 有超过 1000 万开发者、3 亿仓库。从"周末项目"成长为全球最大的代码托管平台。

这样代码有了备份，协作有了中心。

从代码托管到协作平台
---------------------

起初 GitHub 只是个代码托管网站。但随着开源社区的发展，开发者们发现托管只是起点，真正的问题是协作。

假设你发现了一个开源项目的 Bug，想修复它。在 GitHub 出现之前，流程是这样的：

1. 找到作者的邮箱，发邮件说明问题
2. 作者回复说"你修一下发给我"
3. 你修改代码，打包发送
4. 作者手动合并，可能和你本地版本冲突
5. 反复沟通直到合并成功

这个过程可能持续几天甚至几周。GitHub 把它缩短到几小时：你点击 Fork 复制一份仓库，修改后提交 Pull Request，作者在线审查代码，一键合并。

GitHub 从托管平台演化为协作平台，围绕代码托管衍生出一整套协作机制：

- **Pull Request**：在线代码审查和合并
- **Issues**：问题跟踪和讨论
- **Actions**：自动化测试和部署
- **Projects**：任务看板管理

这些功能的设计都指向一个目标：让代码协作像面对面交流一样自然。

Git 与 GitHub 的关系
--------------------

很多初学者混淆 Git 和 GitHub。理解两者的区别是使用 GitHub 的第一步。

Git 是版本控制系统，是一个命令行工具。你在本地执行 ``git commit``，代码变更就记录在本地仓库里。即使没有网络，Git 也能正常工作。

GitHub 是托管平台，是一个网站。它存储你的 Git 仓库，提供 Web 界面和协作功能。推送代码到 GitHub 需要 ``git push`` 命令，这需要网络连接。

用一个类比来理解：Git 像是你的本地笔记本，GitHub 像是云同步服务。

.. code-block:: bash

   # Git 是本地工具，无需网络
   git init                    # 创建本地仓库
   git add README.md           # 添加文件到暂存区
   git commit -m "first"       # 提交到本地仓库

   # GitHub 是远程托管，需要网络
   git remote add origin https://github.com/user/repo.git
   git push -u origin main     # 推送到 GitHub

这段代码展示了完整的流程。前三行命令在本地执行，最后两行连接到 GitHub。如果你在 ``git push`` 前断网，本地仓库依然完好，只是代码暂时不会同步到 GitHub。

为什么选择 GitHub
-----------------

代码托管平台不止 GitHub，还有 GitLab、Bitbucket、Gitee 等。但 GitHub 是最大的开源社区，这带来的优势远超功能层面的比较。

**生态系统优势**

GitHub 拥有超过 1 亿开发者用户和 4 亿仓库。这意味着：

- 开源项目几乎都在 GitHub 上，学习资源丰富
- 第三方工具优先支持 GitHub（如 CI 工具、IDE 插件）
- 招聘时 GitHub 活跃度成为参考指标

**社区效应**

当你在 GitHub 上发布项目，其他开发者可以：

- 通过 Issues 报告 Bug 或提出建议
- 通过 Pull Request 贡献代码
- 通过 Discussions 讨论技术问题

这种开放的协作模式让项目快速成长。TensorFlow、React、Linux 内核等知名项目都在 GitHub 上托管，证明了平台对开源生态的价值。

**功能演进**

GitHub 在 2018 年被微软收购后，功能迭代加速。GitHub Actions 的推出改变了 CI/CD 市场，GitHub Codespaces 提供云端开发环境，GitHub Copilot 引入 AI 辅助编程。这些功能让 GitHub 从托管平台演化为完整的开发平台。

当然，其他平台也有独特优势：GitLab 提供更强的自托管能力，Bitbucket 与 Atlassian 工具集成更好，Gitee 在中国访问更快。选择平台时，除了功能，还要考虑团队需求、地区因素和生态系统。

从哪里开始
----------

如果你刚接触 GitHub，接下来的步骤是：

1. 注册 GitHub 账户（下一节）
2. 配置 SSH 密钥，实现免密推送
3. 克隆一个仓库，体验 Git 操作
4. 创建自己的第一个仓库

这些操作会在后续章节详细介绍。理解 GitHub 的定位——它是 Git 的云端延伸，更是协作的社交平台——能帮助你更好地使用它。
