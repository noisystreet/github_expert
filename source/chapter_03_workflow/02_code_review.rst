代码审查
========

上一节我们了解了 Pull Request 的流程——创建、审查、合并。现在让我们深入审查环节：作为审查者如何给出有效的意见，作为开发者如何回应审查。

代码审查不仅是检查代码正确性，更是知识共享和质量保障的过程。好的审查能发现 Bug、优化设计、传播最佳实践。差的审查只会挑剔格式、敷衍通过，甚至引发冲突。

为什么需要代码审查
-------------------

有人问：既然有自动化测试，为什么还需要人工审查？测试能发现功能错误，但不能发现设计问题、可读性问题、安全隐患。

**审查能发现的问题**

一个真实的例子：开发者写了一段数据库查询代码：

.. code-block:: python

   def get_user(user_id):
       query = f"SELECT * FROM users WHERE id = {user_id}"
       return db.execute(query)

这段代码功能正确，测试也能通过。但审查者一眼看出问题：SQL 注入漏洞。如果 ``user_id`` 是用户输入的 ``"1 OR 1=1"``，查询会返回所有用户数据。

测试不会检查这种情况，但审查者能发现。这就是人工审查的价值：发现测试无法覆盖的问题。

**审查的其他价值**

除了发现 Bug，代码审查还有更多价值：

- **知识共享**：审查者通过阅读代码学习新技巧，开发者通过审查意见学习最佳实践
- **团队共识**：审查过程让团队了解彼此的代码风格和设计思路
- **质量保障**：多人审查比单人审查更能发现问题
- **新人培养**：新人提交代码，老手审查指导，快速成长

Google 的研究表明，代码审查发现的问题中，只有一小部分是功能错误，大部分是设计问题、可读性问题、维护性问题。这些问题会影响长期质量，测试难以发现。

审查者的角色
------------

审查者不是代码的"审判者"，而是"协作者"。审查的目标是帮助开发者改进代码，不是批评开发者。

**审查者的职责**

- 检查代码的正确性和逻辑
- 关注代码的可读性和可维护性
- 发现潜在的性能和安全问题
- 提出建设性的改进建议
- 确认测试覆盖和文档更新

**审查者不应该做的事**

- 拘泥于个人偏好（如变量命名风格）
- 提出与本次改动无关的要求（如"顺便重构其他代码"）
- 使用贬低性语言（如"这代码写得太差了"）
- 拖延审查（一周后才回复）

审查者要记住：你的意见会影响开发者的心情和工作效率。友善、具体的意见比批评更有价值。

审查的要点
----------

打开 PR 的 "Files changed" 页面，你会看到所有改动。审查时应该关注什么？

**第一层：正确性**

最基本的问题：代码是否正确实现了功能？

- 逻辑是否正确？
- 边界条件是否处理？
- 异常情况是否考虑？

如果代码有明显的 Bug，直接指出：

::

   这里的循环条件有误，应该是 i < length 而不是 i <= length，
   否则会访问数组越界。

**第二层：设计**

代码不仅要正确，还要设计合理。

- 代码结构是否清晰？
- 函数职责是否单一？
- 是否有重复代码？
- 是否有更好的实现方式？

设计问题可能需要讨论：

::

   这个函数做了三件事：验证输入、处理数据、保存结果。
   建议拆分成三个函数，职责更清晰。

**第三层：可读性**

代码是写给人看的，要让读者容易理解。

- 变量和函数命名是否清晰？
- 逻辑是否需要注释解释？
- 代码格式是否一致？

可读性问题通常可以提出具体建议：

::

   变量名 `temp` 不够清晰，建议改为 `processed_data`，
   表明这是处理后的数据。

**第四层：性能和安全**

某些代码可能有性能或安全隐患。

- 是否有不必要的循环或查询？
- 是否有内存泄漏风险？
- 是否有安全漏洞？
- 是否有敏感信息暴露？

这些问题需要明确指出风险：

::

   在循环中调用数据库查询会导致 N+1 问题，
   建议先批量查询，再处理数据。

**第五层：测试和文档**

改动代码后，测试和文档是否同步更新？

- 是否添加了新功能的测试？
- 是否更新了 API 文档？
- 是否更新了 README？

提醒开发者补充：

::

   这个新功能缺少测试，建议添加单元测试覆盖主要场景。

审查的优先级：正确性 > 设计 > 安全 > 可读性 > 格式。如果代码有 Bug，格式再美观也没意义。

使用 GitHub 的审查工具
-----------------------

GitHub 提供了多种审查工具，让审查过程更高效。

**行内评论**

点击某行代码的 "+" 按钮，可以添加行内评论。这种评论与具体代码关联，开发者容易理解你指的是哪里。

行内评论适合提出具体问题：

::

   第 45 行：这里的条件判断可以简化为 `if not user.is_active:`

**建议（Suggestions）**

GitHub 支持在评论中直接提供代码建议。开发者可以一键接受建议，自动应用修改。

格式如下：

::

   ```suggestion
   if not user.is_active:
       return None
   ```

开发者点击 "Commit suggestion"，这行代码就会自动修改。省去了手动编辑的步骤。

**多条评论批量提交**

审查完所有文件后，点击 "Review changes" 按钮，提交审查意见。你可以选择：

- **Comment**：提出意见，但不影响合并状态
- **Approve**：批准 PR，可以合并
- **Request changes**：要求修改后才能合并

建议把所有意见整理成一条审查评论，而不是零散提交。这样开发者能看到完整的审查结果，一次性处理。

审查评论示例：

::

   ## 审查意见
   
   整体设计合理，有几个小问题需要处理：
   
   1. 第 45 行：建议简化条件判断（已提供 suggestion）
   2. 第 67 行：数据库查询有 N+1 问题，建议批量查询
   3. 缺少单元测试，建议补充
   
   修改完成后我会批准合并。

这种结构化的评论比零散评论更有价值。

处理审查意见
-------------

作为开发者，收到审查意见后如何处理？

**第一步：理解意见**

不要急于反驳，先理解审查者的意图。审查者提出意见是为了改进代码，不是找茬。

如果意见不够清晰，可以追问：

::

   你提到这里的逻辑有问题，能具体说明是哪种情况会出错吗？

**第二步：分类处理**

审查意见可以分为三类：

- **必须修改**：明显的 Bug、安全问题、设计缺陷
- **建议修改**：可读性改进、性能优化、代码重构
- **可以讨论**：风格偏好、可选方案

必须修改的问题一定要处理。建议修改的问题权衡成本收益。可以讨论的问题根据实际情况决定。

**第三步：修改代码**

根据意见修改代码：

.. code-block:: bash

   # 在 feature 分支上修改
   git checkout feature/user-profile
   
   # 应用审查意见
   git add .
   git commit -m "Address review feedback: simplify condition, fix N+1 query"
   
   # 推送更新
   git push origin feature/user-profile

推送后 PR 自动更新，审查者可以看到修改。

**第四步：回应意见**

修改完成后，在审查评论下回复：

::

   已修改：
   1. 第 45 行：应用了 suggestion
   2. 第 67 行：改用批量查询
   3. 添加了单元测试
   
   请再次审查。

如果某些意见你不认同，解释原因：

::

   关于变量命名，我使用 `temp` 是因为这个变量只在这个函数内部使用，
   作用范围有限，不需要更长的名称。

礼貌地解释，不要对抗。审查者通常会接受合理的解释。

审查清单：不遗漏检查项
------------------------

审查时容易遗漏某些检查项。一个清单可以帮助审查者系统性地检查。

**审查清单示例**

.. list-table:: 代码审查清单
   :widths: 30 70
   :header-rows: 1

   * - 检查项
     - 说明
   * - 功能完整性
     - 是否实现了 PR 描述的功能
   * - 逻辑正确性
     - 边界条件、异常处理是否正确
   * - 代码可读性
     - 命名、注释、格式是否清晰
   * - 设计合理性
     - 结构、职责、复用是否合理
   * - 测试覆盖
     - 是否有足够的测试
   * - 文档更新
     - 是否更新了相关文档
   * - 性能影响
     - 是否有性能问题
   * - 安全风险
     - 是否有安全漏洞

审查者可以按清单逐项检查，确保不遗漏。

团队可以定制清单，添加项目特有的检查项。比如涉及数据库改动的 PR，检查项可以包括"是否通知 DBA"、"是否更新数据库迁移脚本"。

审查文化：让审查有价值
------------------------

代码审查的价值取决于团队文化。好的审查文化让审查成为学习和协作的机会，差的审查文化让审查成为形式主义。

**建立健康的审查文化**

**友善的语气**

审查意见应该友善、具体。比较两种表达：

- 差："这代码写得太差了，重写"
- 好："这个函数的职责有点复杂，建议拆分成两个函数"

第二种表达指出了问题，提供了建议，语气友善。开发者更愿意接受。

**聚焦代码，不是个人**

审查时避免使用"你"这个字：

- 差："你写的这段代码有 Bug"
- 好："这段代码在边界情况下有 Bug"

聚焦代码本身，避免指向个人，减少防御心理。

**及时审查**

PR 创建后尽快审查。拖延审查会影响开发者的进度。建议在 24 小时内完成审查。

**审查者与开发者互相学习**

审查不是单向的。审查者通过阅读代码学习新技巧，开发者通过审查意见学习最佳实践。鼓励双方提问和讨论。

**避免过度审查**

有些审查者过于细致，对每个细节都提意见。这会让开发者疲惫，降低审查效率。

审查应该聚焦重要问题：正确性、设计、安全。格式、风格的小问题可以忽略，或用自动化工具处理（如 lint 工具）。

从审查到持续改进
-----------------

代码审查不是终点，而是持续改进的过程。每次审查后，团队可以反思：

- 常见问题是什么？
- 如何避免类似问题？
- 审查流程是否高效？

根据反思结果调整审查清单、改进开发流程、引入自动化工具。
