Pull Request
============

上一章我们理解了分支策略——用分支隔离开发，避免互相干扰。现在的问题是：分支开发完成后，如何安全地合并到主分支？

直接用 ``git merge`` 合并看似简单，但存在风险。假设你在 feature 分支上开发了一个新功能，合并到 main 后发现引入了 Bug。这时主分支已经损坏，影响其他开发者。如果团队有十个人，每个人都这样合并，main 分支会变得不稳定。

Pull Request（PR）解决了这个问题。它在合并前设置一个检查点：其他人审查你的代码，确认无误后才允许合并。这是一个协作机制，把"合并"变成"讨论+审查+合并"的完整流程。

Pull Request 的本质
-------------------

Pull Request 不是 Git 的概念，而是 GitHub 的协作功能。它的名字容易让人困惑：为什么叫 "Pull" 而不是 "Push"？

理解这个名字需要换个视角。你是分支的开发者，想把自己的代码"推送"到主分支。但从主分支维护者的视角，他在"拉取"你的改动，决定是否接受。

所以 Pull Request 的含义是：请求主分支"拉取"你的改动。你发起请求，维护者审查后决定是否接受。

**PR 的工作流程**

一个完整的 PR 流程：

1. 开发者创建功能分支，开发新功能
2. 推送分支到 GitHub
3. 在 GitHub 上创建 Pull Request
4. 审查者查看改动，提出意见
5. 开发者根据意见修改代码
6. 审查者批准，合并 PR
7. 分支自动删除（可选）

这个过程把合并变成了协作，而不是一个人的独断决策。

创建 Pull Request
-----------------

**第一步：准备分支**

从 main 分支创建一个功能分支：

.. code-block:: bash

   # 确保本地 main 分支是最新的
   git checkout main
   git pull origin main

   # 创建功能分支
   git checkout -b feature/user-profile

   # 开发功能，提交代码
   git add .
   git commit -m "Add user profile page"

   # 推送分支到 GitHub
   git push -u origin feature/user-profile

推送后，GitHub 会显示一个提示："feature/user-profile had recent pushes"。点击 "Compare & pull request" 按钮，开始创建 PR。

**第二步：填写 PR 信息**

PR 页面需要填写几个关键信息：

- **标题**：简洁描述这个 PR 做了什么，如 "Add user profile page"
- **描述**：详细说明改动内容、背景、测试方法
- **目标分支**：选择要合并到哪个分支（默认是 main）
- **审查者**：指定谁审查这个 PR

一个好的描述应该包含：

::

   ## 改动内容
   - 新增用户个人主页
   - 添加头像上传功能
   - 支持个人信息编辑

   ## 测试方法
   1. 启动应用：`python app.py`
   2. 登录用户账号
   3. 访问 `/profile` 页面
   4. 测试头像上传和信息编辑

   ## 相关 Issue
   Fixes #123

**第三步：创建 PR**

点击 "Create pull request"，PR 就创建完成了。你会看到一个 PR 页面，显示：

- 改动的文件列表
- 每个文件的改动详情（diff）
- 审查者的状态
- CI 测试的状态（如果有）

PR 的审查过程
-------------

创建 PR 后，审查者会收到通知。审查者打开 PR 页面，查看改动。

**查看改动**

PR 页面的 "Files changed" 标签显示所有改动。绿色行是新增内容，红色行是删除内容。审查者可以逐行查看，理解每个改动的意图。

**添加评论**

如果审查者对某行代码有疑问，可以点击行号，添加评论：

::

   这里的参数验证不够完整，建议添加 email 格式检查。

开发者会收到通知，查看评论后修改代码。

**提交审查意见**

审查者可以提交三种审查意见：

- **Comment**：普通评论，不影响合并
- **Approve**：批准，允许合并
- **Request changes**：要求修改，修改后才能合并

如果审查者选择了 "Request changes"，PR 会标记为"需要修改"。开发者必须回应这些意见，修改代码或解释原因。

**修改代码**

开发者根据审查意见修改代码：

.. code-block:: bash

   # 在 feature 分支上修改代码
   git checkout feature/user-profile

   # 修改文件
   git add .
   git commit -m "Add email format validation"

   # 推送更新
   git push origin feature/user-profile

推送后，PR 自动更新，显示新的改动。审查者可以看到你的修改，继续审查。

这个循环持续直到审查者批准。

合并 Pull Request
-----------------

审查通过后，可以合并 PR。GitHub 提供三种合并方式：

.. list-table:: 合并方式对比
   :widths: 20 40 40
   :header-rows: 1

   * - 方式
     - 效果
     - 适用场景
   * - Merge commit
     - 创建合并提交，保留分支历史
     - 需要追踪每个 PR 的大型项目
   * - Squash and merge
     - 把所有提交压缩成一个
     - 保持 main 历史整洁
   * - Rebase and merge
     - 把提交重新应用到 main，线性历史
     - 需要线性历史的项目

**推荐使用 Squash and merge**

功能分支上的提交通常是碎片化的："fix typo"、"update variable"、"add test"。这些提交单独看意义不大，合并到 main 后会让历史变得混乱。

Squash and merge 把这些提交压缩成一个，标题使用 PR 的标题。这样 main 分支的每个提交都是一个完整的功能，历史清晰易读。

点击 "Merge pull request"，选择合并方式，确认合并。PR 合并后，GitHub 会提示删除分支。如果分支不再需要，建议删除，减少分支列表的混乱。

PR 模板：规范协作流程
----------------------

团队协作时，每个 PR 应该包含一致的信息：改动内容、测试方法、相关 Issue。如果开发者随意填写，审查者难以理解改动。

PR 模板解决了这个问题。在仓库根目录创建 ``.github/pull_request_template.md``，每次创建 PR 时，GitHub 会自动填充模板内容。

一个实用的 PR 模板：

::

   ## 改动描述
   
   简要描述这个 PR 的改动内容。

   ## 改动类型
   
   - [ ] 新功能
   - [ ] Bug 修复
   - [ ] 重构
   - [ ] 文档更新

   ## 测试方法
   
   描述如何测试这些改动。

   ## 检查清单
   
   - [ ] 代码通过了 lint 检查
   - [ ] 添加了必要的测试
   - [ ] 更新了相关文档
   - [ ] 关联了相关 Issue

   ## 相关 Issue
   
   Fixes #

创建 PR 时，模板内容会出现在描述框中，开发者填写后提交。这确保每个 PR 都有完整的信息，减少审查者的困惑。

PR 最佳实践
-----------

**保持 PR 小而专注**

一个 PR 应该只做一件事。如果 PR 改动了十个文件，涉及三个功能，审查者很难理解每个改动。建议每个 PR 控制在 400 行改动以内。

如果改动太大，拆分成多个 PR：

- PR 1：重构代码结构
- PR 2：添加新功能
- PR 3：更新文档

这样每个 PR 目标明确，审查效率更高。

**编写清晰的描述**

PR 描述是审查者理解改动的关键信息。不要只写"修复 Bug"或"添加功能"，要具体说明：

- 修复了什么 Bug，如何修复
- 添加了什么功能，为什么需要
- 改动的背景和影响范围

**添加必要的测试**

改动代码后，应该添加对应的测试。这证明改动正确工作，也防止未来修改破坏功能。

PR 模板中可以要求开发者勾选"添加了测试"，作为合并的前提条件。

**及时响应审查意见**

审查者提出意见后，尽快回应。如果意见合理，修改代码；如果不合理，解释原因。

长时间不回应会让审查者等待，拖慢项目进度。建议在 24 小时内回应审查意见。

从 PR 到协作文化
-----------------

Pull Request 不仅是技术工具，更是协作文化的体现。它把"合并代码"变成"讨论代码"，把个人决策变成团队决策。

一个健康的 PR 文化：

- 审查者友善地提出意见，不是批评
- 开发者虚心接受意见，不是抵触
- 讨论聚焦于代码，不是个人
- 小 PR 频繁合并，不是大 PR 长期停滞

这种文化让代码质量提升，也让团队关系融洽。
